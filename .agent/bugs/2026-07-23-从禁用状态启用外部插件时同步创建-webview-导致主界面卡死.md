# 2026-07-23 外部插件窗口同步在主线程创建 WebView 导致应用卡死

## 现象

两种路径都可能让整个应用进入“未响应”：

1. `demo-timer` 处于禁用状态，在插件页开启它；
2. 上一次已启用插件，重新启动应用后，前端首次扫描插件时直接卡死。

日志通常停在第二次 `scanned 1 plugin(s)`，看不到 `started background window`。

## 根因

插件窗口同步包含 `WebviewWindowBuilder::build()` / `window.destroy()`。最初这些生命周期操作直接在 invoke 中同步执行，会阻塞开关调用。

第一次修复错误地改成了 `AppHandle::run_on_main_thread()`：这反而把整个 `sync()`（包括 `WebviewWindowBuilder::build()`）放进 Tauri 主线程。Windows 下 `build()` 自身需要与主事件循环协作；从主线程任务里同步等待它会形成自阻塞。因此前端首次 `list_external_plugins` 排队同步后，应用马上“未响应”。

启动 `setup` 中直接调用 `sync()` 也有同样风险，不能在 setup 尚未返回时同步创建插件 WebView。

## 修复

- `PluginWindowManager::schedule_sync()` 使用 `tauri::async_runtime::spawn_blocking()` 执行窗口同步；
- `list_external_plugins` 和 `set_external_plugin_enabled` 更新状态后只投递任务，不等待窗口创建完成；
- 启动 `setup` 也只投递初始同步，先让 Tauri 主事件循环完成启动；
- 保留 `sync_lock`，串行化启动、扫描和开关可能产生的重复同步。

不要把包含 `WebviewWindowBuilder::build()` 的整段同步逻辑放进 `run_on_main_thread()`。

## 验证

- `cargo check`
- `cargo test plugin_ --lib`（4 passed）
- `vue-tsc --noEmit`
- `git diff --check`

手测：

1. 插件保持启用并重启应用，主界面应保持响应；
2. 禁用再启用 `demo-timer`，开关调用应立即完成；
3. 日志随后出现后台窗口启动和脚本加载；
4. 约 10 秒后显示插件 Toast。
