# 插件异常标签如何判定和保持不拦截

插件中心的“异常”标签是诊断信号，不是权限系统或资源限额。宿主发现某个外部插件持续占用资源或产生异常数据活动后，记录插件 id、写 warning，并让 Plugins 页面显示标签；插件调用仍继续执行。

## 数据流

```text
plugin-bg-<id>
  ├─ plugin_publish_event ──┐
  ├─ plugin_storage_set ────┼─> PluginResourceActivity（按 plugin id、60 秒窗口）
  └─ plugin_report_memory ──┘
                                ├─ mark_anomalous(id)
                                ├─ log warning（含原因和 plugin id）
                                └─ emit catrace:plugin-anomaly
                                         └─ Plugins.vue 实时显示“异常” Tag
```

## 当前判定

| 类型 | 判定方式 | 数据来源 |
|------|----------|----------|
| 刷事件 | 60 秒内发布超过 60 个 Event | `plugin_publish_event` |
| 单次大 Event | Event 序列化后达到 8 MiB | `plugin_publish_event` |
| 连续磁盘写入 | 60 秒内通过插件 Storage 写入超过 16 MiB | `plugin_storage_set` |
| 单次大 Storage value | 单次序列化 value 达到 8 MiB | `plugin_storage_set` |
| 连续高内存 | JS heap 每 15 秒采样；连续 4 次达到 128 MiB | `PluginHost.vue` → `plugin_report_memory` |

`performance.memory` 不是所有 WebView 都支持；不支持时跳过内存采样，不影响插件运行。这里观测的是插件后台 WebView 的 JS heap，不等同于操作系统进程完整 working set。

## 必须保持的约定

- 判定阈值只用于**标记和诊断**，不得用于拒绝、截断、限流或丢弃调用。
- Event、Storage value 和日志仍不设置隐藏输入大小限制。
- warning 必须包含 plugin id 和异常原因，方便用户定位并修复对应插件。
- 异常状态由 `PluginManager` 在内存中保存，插件列表 rescan 时按 id 保留；应用重启后清除。
- `catrace:plugin-anomaly` 用于页面实时更新；即使页面未监听，下一次读取插件列表也能取得 `anomalous: true`。
- 当前 Tag 只表示“本次运行中观察到异常”，不区分具体原因；具体原因查日志。

## 修改判定时要改哪些地方

1. `src-tauri/src/plugin_commands.rs`：资源活动窗口、阈值、原因日志和单元测试。
2. `src/views/PluginHost.vue`：内存采样频率和 WebView 能力降级。
3. `src-tauri/src/plugins.rs`：异常状态保存与 rescan 保留逻辑。
4. `src/views/Plugins.vue`：实时事件监听和列表/详情 Tag。
5. `src/api/tauri.ts`：`ExternalPluginInfo.anomalous` 类型。
6. `.agent/architecture/desktop-event-os/step3-roadmap-plugin-runtime.md`：更新当前真源。
