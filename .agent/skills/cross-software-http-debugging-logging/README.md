# 跨软件 HTTP 联调：开发初期就写详细日志

一条跨项目通用的调试纪律：**当你的程序要接收另一个软件（CLI / agent / 第三方服务）发来的 HTTP 请求时，从写接收端的第一天起，就在请求入口打详细日志。** 不要等「联调不通」再回头补。

> 源自 Catrace P6 权限审批联调（2026-07-18）：一个「请求缺字段导致反序列化失败」的 bug，因入口无日志被掩盖成「请求根本没到」，排查了一下午。见 `.agent/bugs/2026-07-18-permission-card-not-popping-event-field-missing.md`。

## 为什么

跨软件联调的失败模式里，**「对端到底发没发、发了什么」是最常见、也最难隔岸观察的一环**：

- 对端是黑盒（别人的 CLI / 系统），你控制不了它的日志。
- 你的接收端如果「解析失败 → 静默 return」，现象和「请求根本没到」**一模一样**。
- 没有入口日志时，你会去怀疑一万个别的东西（端口、配置、进程、网络、时序），最后才发现是自己把它拒了。

入口两行日志能把「是不是我的问题」一秒钟切开：

```
>>> 收到 POST /permission            ← 到没到，一眼可见
    body = {...}                     ← 发的什么，字段全不全
    解析失败: missing field `event`   ← 是对方格式问题还是我的 schema 问题
```

## 怎么做（接收端清单）

写一个会被外部调用的 HTTP 端点时，**第一版就带上**：

1. **请求入口日志**：方法 + 路径 + （脱敏后的）body。哪怕先用 `println!` / 临时落盘文件。
2. **解析失败必打原因 + 原始 body**：绝不 `Err(_) => 静默 400`。外部输入的格式假设几乎总会被真实请求打破。
3. **分支走向日志**：进了哪个处理分支、为什么提前 return（开关关着？模式 off？去重？）。
4. **跨线程/跨进程边界各打一端**：发送方「我发了」、接收方「我收到了」，两端对上才算通。

## 配套纪律

- **测试样例逐字段对齐真实请求**——包括真实请求**缺少**的字段。手搓一个「看着像」的 body，会让「必填字段缺失」这类 bug 在测试里隐形、在真机才炸。
- **反序列化外部输入，字段默认 `#[serde(default)]` / optional**；只有协议硬性保证必发的字段才敢设必填。
- **临时日志记得收口**：联调通过后，把临时的高频/落盘日志降级或删除，别留在正式代码里刷盘。
- **双向都模拟一遍**：不仅能「我用 curl 打自己」，还要能「让真实对端打我」——两者都通才算联调完成。

## 反模式（都踩过）

- 「先跑通再说，日志后面补」→ 联调不通时没有任何线索，只能猜。
- 「解析失败就 return 400，反正对端会重试」→ 对端不重试，且你完全不知道发生了什么。
- 「测试 body 我自己编一个」→ 和真实请求不一致，bug 被测试「通过」掩盖。
