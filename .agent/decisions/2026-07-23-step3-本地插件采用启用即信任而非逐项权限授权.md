# 2026-07-23 Step3 本地插件采用启用即信任而非逐项权限授权

## 背景

Step 3 M11 已实现插件后台隐藏 WebView 和 `plugin_*` invoke。首版 manifest 包含 `permissions`，Rust 在 notification、activity、storage、logger、clipboard 等命令入口检查插件是否显式声明对应权限。

当前插件来源是用户手动放入本机插件目录并主动启用，没有插件市场、远程静默安装或安装时权限确认页面。插件本身又能在 WebView 中使用 `fetch` / `WebSocket`。在这种前提下，插件作者可以自行把所有权限写进 manifest，单纯的声明字符串不能形成真正的用户授权边界，却会增加插件开发和使用摩擦。

VS Code、uTools、Rubick 等桌面插件系统的主要信任动作也是“安装/启用插件”，而不是为 notification、storage、clipboard 等普通能力逐项授权。它们仍保留运行环境隔离、对象归属、路径限制和资源保护。

## 决策

Catrace Step 3 采用 **Trusted Local Plugin Model（本地插件启用即信任）**：

- 用户手动安装并启用插件，即视为允许其调用 Catrace 已公开的普通宿主 API。
- 普通 `plugin_*` invoke 不再要求 manifest 声明对应 `permissions`，也不建设逐项授权弹窗、“允许一次/永久允许”等流程。
- `permissions` 若暂时保留，只作为插件详情页的能力说明元数据，不作为运行时拒绝依据。
- 真正高风险且当前没有明确需求的能力优先选择“不开放”，而不是开放后依赖插件自行声明权限。

## 必须保留的强制边界

取消 manifest 权限硬门闩不等于取消宿主边界。Rust 仍必须强制：

1. **调用方身份**：后台插件 id 只能从 `plugin-bg-<id>` 窗口 label 推导，不能相信插件自报 id。
2. **启用状态**：插件禁用或失效后，残留窗口/调用不得继续使用宿主能力。
3. **Event 所有权**：Event source 由宿主填写；保留 `allows_event`、保留 kind 拒绝和跨插件 Event 校验。
4. **私有命名空间**：Storage 与未来文件目录按 plugin id 隔离，不能读写其他插件数据或越界路径。
5. **输入边界**：Storage、剪贴板、Event payload、日志等保留大小、格式和合法值校验。
6. **防失控**：publish 保护按插件隔离，正常提醒无感，只处理明显突发或持续失控。

## M12 能力原则

- `plugin_open` 使用协议/目标白名单，不接受任意 shell 字符串。
- 前台窗口能力只返回必要的结构化快照，不暴露键序列和坐标。
- 文件能力只允许插件自己的数据目录，路径由 Rust 归一化并验证。
- 读剪贴板、任意文件访问、任意 shell/进程执行在没有明确需求前不开放。

## 影响

- M11.1 从“资源与权限收紧”改为“资源与隔离边界”。
- 现有 `require_permission` / `has_permission` 硬门闩将在后续代码迭代中移除或改为“已启用插件身份检查”。
- 调用方身份、Event source、Storage namespace、输入限制等现有检查继续保留。
- publish 不直接照搬 HTTP SDK 的严格阈值，具体策略以“正常无感、明显失控才介入”为目标，通过真机体验确定。
