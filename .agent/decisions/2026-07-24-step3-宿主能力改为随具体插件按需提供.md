# Step 3 宿主能力改为随具体插件按需提供

## Status

Accepted

## Context

Step 3 M11/M11.1 已完成插件后台运行时、基础 invoke、身份与所有权边界，以及异常活动观测。原路线图把 `plugin_open`、前台窗口快照、受限写文件和 napcat 示例统一列为下一阶段 M12。

当前没有具体插件立即依赖这些能力。若先建设通用 API，容易在缺少真实调用场景时扩大宿主接口和平台适配范围。

## Decision

不再把 M12 宿主能力作为独立的前置开发阶段。后续做到具体插件时：

1. 先确认插件的真实交互需求。
2. 只提供该插件需要的最小宿主能力。
3. 能由 WebView 原生 `fetch` / `WebSocket` 完成的网络访问，不增加宿主代理。
4. 新增宿主能力时，同时在 Rust 固定调用方身份、协议、目标、路径或对象所有权边界。
5. 不因为路线图列出了候选能力，就一次性补齐未被使用的通用 API。

## Consequences

- Step 3 基础 Plugin Runtime 以 M11.1 为当前完成点。
- `plugin_open`、`plugin_get_active_window`、`plugin_write_file` 均没有预定的立即开发顺序。
- napcat 链路等到对应消息插件开发时再做端到端验证。
- 每个新能力的设计和验收标准由实际插件场景驱动。
